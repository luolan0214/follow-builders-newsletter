Theme 05

Podcast: Granola CEO — Knife Fights & AI-Native Work / 播客：Granola CEO——刀战与 AI 原生工作

AI & I (Every) × Chris Pedregal（Granola CEO）。完整中文译文：15 亿美元估值的会议笔记公司如何思考竞争、AI 原生工作界面、pre-generation 策略、token 经济学、以及为什么创业即使成功也是刀战。

A&
AI & I by Every
Dan Shipper 主持 × Chris Pedregal（Granola CEO）
中文

Granola CEO Chris Pedregal（15 亿美元估值，60-70 人团队）做客 AI & I 与 Dan Shipper 对谈。核心话题：（1）创业就是刀战——不管顺不顺利。（2）会议笔记不是终极价值——真正的奖品是发明 AI 原生工作界面。（3）Dan 的「海盗/架构师」产品团队框架。（4）预生成策略：Granola 预生成数百万份会前简报，只有约 10% 被打开，但在需要时至关重要。（5）「Codex 原生应用」范式——把你的 agent 带到应用里。（6）Token 支出追踪直到上周才开始，触发因素是 Fable。（7）多上下文 agent 工作缺失的 UI 范式。（8）每周约一半 Granola 用户使用 agentic 功能。

【创业就是刀战——即使成功也是】

Chris Pedregal 说：创业就像刀战。我以前以为只有不顺利的时候才难。结果顺利的时候一样难。

「我做了一家之前的创业公司 Socratic，12 个人。从 12 人到现在的 60-70 人，完全不同，我从来没做过这件事。组织人类——这是一件很难的事。」

【会议笔记不是终极价值】

Pedregal 说：会议笔记不是大家追逐的终极价值。有更大的东西——我们用什么界面工作、在 AI 原生世界里我们怎么工作。

「我们不是第一个会议笔记工具。AI 笔记工具存在很久了。我们受到之前很多东西的启发。所以我不能说『你怎么能受我们启发』——这是一个生态系统。」

「Notion 做了类似的东西，OpenAI 做了，Zoom 最近也做了。在某些方面我不需要去经历那个噩梦般的假设场景——它来了，发生了，我们还在。增长 rate 没变。」

【Dan 的「海盗/架构师」框架】

Dan Shipper（Every CEO）说：对于早期产品，有两个角色重要。一个是海盗——以最快速度建造，找到有价值的东西，可以 vibe code，不用想架构。另一个是架构师——与海盗配对，当海盗发现有价值的东西时，思考如何把它变成可持续、可扩展的系统。

「每个产品需要一个海盗。但一个架构师可以服务多个产品——因为现在的工具太好了，一个优秀的程序员可以用 Claude 或 Codex 在一小时内学会一个代码库。」

【预生成策略——扶手栏杆哲学】

Pedregal 说：如果让 Granola 做事要等 20 秒，人类在忙碌的会议日里几乎不会等。但如果 Granola 预先生成你可能需要的东西，然后说『嘿，如果你需要这个，它已经在这里了』——感觉完全不同。

「我们预生成数百万份会前简报。只有大约 10% 被打开。但在你打开的那一刻——你真的非常感激，因为那正是你需要的。」

Granola 的产品哲学比喻：扶手栏杆。「所有楼梯都有栏杆。你从不注意到它。但当你绊倒的那一刻，你的手伸出去——它需要在那里，需要能承重。这就是我们对 Granola 的想法。」

【「Codex 原生应用」——把 agent 带到应用里】

Dan Shipper 说：我用 Codex 里几乎所有的软件都在应用内浏览器里。我把 Codex 带到我的邮件或任何其他东西里。

「昨晚我搬了家，需要改网络。我告诉 Codex 去搞定。它在应用内浏览器里打开了 Verizon 网站，登录了，然后跟客服 agent 聊天——客服可能也在用 AI——直到把我的网络切换好。」

Pedregal 的回应：如果 MCP 有状态化的 UI——agent 通过 MCP 修改 UI 的实时状态——用户和 agent 看着同一个界面，那会很有意思。

【Token 经济学——Fable 触发的觉醒】

Pedregal 说：我们直到上周才开始追踪 token 支出。触发因素是 Fable——有人说『我今天可能花了几千美元』，我们想『好吧，也许我们应该至少知道花了多少。』

Dan Shipper：我们一模一样。Fable 出来后，运行 Quora 的 Kieran 快要在 credits 上花 100 万美元了。

「在产品中，我仍然相信现在是早期阶段——我们应该搞清楚什么是杀手级 AI 原生体验，因此我们在 token 支出上不太节约。成本会随时间下降，等我们知道了最佳体验是什么再优化。」

【缺失的 UI 范式 + 50% 用户使用 agentic 功能】

Pedregal 说：当你有多场会议的上下文时，正确的 UI 和交互设计是什么？没有人想清楚了。聊天 UI 出奇地多功能，但当你跨多个上下文做多件事时，我们没有找到正确的隐喻。

「每周大约一半的 Granola 用户以 agentic 方式使用我们——他们不会这么描述，但他们会问一个复杂的查询，比如跨多场会议做分析。大概一半。」
