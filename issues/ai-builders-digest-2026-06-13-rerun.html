Does Git Need to Die? & Agent Orchestration / Git 该死了吗？与 Agent 编排

S
Swyx
Writer / Builder
@swyx
中文

swyx 的挑衅性论点：PR 死了、Code Review 死了，Git 是不是也该死了？大约 20-40% 的代码花费在管理和解决 merge conflicts 上。「我们和人类同事合作时不做逐行 merge conflict——而是聊天、建议修改、旁边评论，然后负责人 ship。」也许未来的代码库更像 Notion 或 Linear 数据库而非 .git 对象。「效率更低，但扩展性更好。正是 Salty Lesson。」90 赞。

这是对版本控制基本单元的直接挑战。如果 agent 自主产出和审查代码，merge-conflict 抽象可能已经过时了。

swyx：PR 死了 Code Review 死了 Git 也该死？20-40%代码花在 merge conflicts。未来代码库更像 Notion/Linear。90 赞。

English

swyx's provocative thesis: after PRs die and Code Review dies, does Git need to die next? Roughly 20-40% of code spend is just managing merge conflicts. 'We don't do line by line merge conflicts when we collaborate with human colleagues—instead we chat, suggest edits, do side comments, and an owner ships it.' Maybe the future codebase looks more like a Notion or Linear database than .git objects. 'It will be less efficient, but more scalable. Exactly the Salty Lesson.' 90 likes.

This is a direct challenge to the fundamental unit of version control. If agents produce and review code autonomously, the merge-conflict abstraction may be obsolete.

After the PR dies, after the Code Review dies, I am seriously wondering if Git needs to die next. Roughly 20-40% of code spend is just managing and updating merge conflicts. We don't do line by line merge conflicts when we collaborate with human colleagues—instead we chat, suggest edits, and an owner ships it. Maybe the future codebase looks more like a Notion or Linear database than .git objects.
